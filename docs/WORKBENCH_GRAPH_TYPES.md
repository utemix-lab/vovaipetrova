# Workbench Graph Types — Графовые эпистемологии

**Дата:** 17 февраля 2026
**Статус:** Ключевой концептуальный документ
**Track:** 1.5 (между Ontology и Schema) — **Epistemic Ontology Layer**

---

## Суть проблемы

Платформа — это инфраструктура для **разных типов миров**.

Мир `vovaipetrova` — демонстрационная среда, которая показывает:
> Может ли одна инфраструктура выдержать разные эпистемологии?

Каждый воркбенч — это не просто "область знаний".
**Воркбенч — это тип онтологии.**

---

## Критический принцип

```
Воркбенчи НЕ ДОЛЖНЫ быть унифицированы.
Их различие — это и есть демонстрация.
```

Если все воркбенчи сводятся к "просто другим данным в одной схеме" — демонстрации не будет.

---

## Типы графовых структур

### Тип A: Фасетная онтология (Faceted Ontology)

**Характеристика:**
- Высоко-формализуемый
- Множество осей классификации
- Атрибутивный
- Структурный

**Узлы:** Сущности + Атрибуты + Оси классификации
**Связи:** `HAS_VALUE`, `CLASSIFIED_AS`, `BELONGS_TO_AXIS`
**Особенность:** Одна сущность принадлежит многим категориям одновременно

**Что проверяет в платформе:**
- Мульти-мерность
- Пересечение осей
- Фильтрация по комбинации атрибутов

---

### Тип B: Граф влияний (Influence Network)

**Характеристика:**
- Исторический
- Временной
- Сетевой
- Культурные связи

**Узлы:** Агенты + События + Эпохи + Жанры
**Связи:** `INFLUENCED`, `CONTEMPORARY_OF`, `BELONGS_TO_ERA`, `EVOLVED_FROM`
**Особенность:** Направленность связей, временная ось

**Что проверяет в платформе:**
- Временные связи
- Направленные графы
- Сетевой анализ (центральность, кластеры)

---

### Тип C: Концептуальный гиперграф (Conceptual Hypergraph)

**Характеристика:**
- Исследовательский
- Неустойчивый
- Допускает противоречия
- Возможны гиперсвязи (связь между >2 узлами)

**Узлы:** Идеи + Практики + Технологии + Эксперименты
**Связи:** `EVOLVES_INTO`, `EXPERIMENT_WITH`, `CONTRADICTS`, `SYNTHESIZES`
**Особенность:** Нестрогая структура, допустимы противоречия

**Что проверяет в платформе:**
- Гибкость модели
- Работа с неопределённостью
- Гиперграфы

---

### Тип D: Процессный граф (Process Graph)

**Характеристика:**
- Workflow-ориентированный
- Этапы и переходы
- Состояния и трансформации

**Узлы:** Состояния + Действия + Артефакты
**Связи:** `TRANSFORMS_TO`, `REQUIRES`, `PRODUCES`, `ENABLES`
**Особенность:** Ориентирован на процесс, не на статику

**Что проверяет в платформе:**
- Динамические графы
- Workflow-логика
- Зависимости

---

### Тип E: Таксономический граф (Taxonomic Graph)

**Характеристика:**
- Иерархический
- Строгое дерево с возможными cross-links
- Научная классификация

**Узлы:** Таксоны разных уровней
**Связи:** `IS_A`, `PART_OF`, `SUBTYPE_OF`
**Особенность:** Строгая иерархия, наследование свойств

**Что проверяет в платформе:**
- Иерархические структуры
- Наследование
- Таксономическая логика

## Персонаж как мост

Персонаж (Character) — это **точка пересечения разных эпистемологий**.

```
Character
    │
    ├── operates_in → Workbench A (Faceted)
    ├── operates_in → Workbench B (Influence)
    └── operates_in → Workbench C (Conceptual)
```

Персонаж НЕ унифицирует воркбенчи.
Персонаж СВЯЗЫВАЕТ их через свои компетенции.

## Коллаб как взаимодействие эпистемологий

**Открытый вопрос:** Что такое Collab в парадигме разных графов?

### Гипотеза 1: Коллаб = Интерфейс между эпистемологиями

Два персонажа с разными типами воркбенчей создают коллаб.
Коллаб — это **зона перевода** между их графовыми структурами.

```
Character A (Faceted) ←→ Collab ←→ Character B (Influence)
```

### Гипотеза 2: Коллаб = Новая эпистемология

Коллаб создаёт **третий тип графа**, который не сводится к исходным.
Это синтез, а не пересечение.

### Гипотеза 3: Коллаб = Проекция пересечения

Коллаб не создаёт новую структуру.
Он показывает **общие узлы** в разных графах через разные линзы.

**Требует дальнейшей проработки.**

## Критические запреты

| Запрет | Почему |
|--------|--------|
| Унифицировать воркбенчи | Убьёт демонстрацию |
| Приводить к одной схеме | Потеря разнообразия |
| Выравнивать классификации | Смешение эпистемологий |
| Требовать совместимости типов | Ограничит платформу |

---

## Что это даёт

1. **Чёткое разделение** — каждый воркбенч знает свой тип
2. **Документальная поддержка** — не сбиться с цели
3. **Архитектурные решения** — вытекают из типа графа
4. **Защита от унификации** — явный запрет
